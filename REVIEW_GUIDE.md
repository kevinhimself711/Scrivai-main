# Scrivai Review Guide

> 最后更新：2026-03-17
> 维护者：Scrivai Lead + 承璋

---

## 1. 项目目标与核心要求

### 项目目标
构建可配置的通用文档生成与审核框架（Python 库），使不同横向项目（南网施工方案、招采审核等）能通过配置文件快速接入，复用统一的生成引擎和审核引擎。

### 核心质量要求（替代"核心假设"）
- **要求 1**: 生成的长文档（8-10章）必须保证术语一致、数据引用不矛盾、上下文连贯 ← 由生成引擎集成测试验证
- **要求 2**: 审核引擎必须能发现文档中的结构缺失、引用失效、内容违规、内部矛盾 ← 由审核引擎集成测试验证
- **要求 3**: 新项目接入只需编写配置文件（YAML + 模板），不需要改框架代码 ← 由多项目 E2E 测试验证

> Reviewer 审查时，必须检查代码变更是否会破坏以上三个核心要求。

## 2. 关键算法与机制

> Reviewer 必须对照此节检查代码实现是否正确。

### 机制 1: 逐章生成 + 上下文管理
- **位置**: `core/generation/engine.py` + `core/generation/context.py`
- **核心流程**:
  ```
  for chapter in chapters:
    1. 检索案例库 → 召回本章相关历史片段
    2. 组装上下文 = 全局术语表 + 前文摘要 + 本章规格 + 检索结果 + 用户输入
    3. LLM 生成本章
    4. 更新术语表 + 交叉引用记录
  ```
- **常见实现错误**:
  - 前文摘要未随章节推进更新，导致后续章节上下文过时
  - 术语表未去重或冲突检测，同一概念出现不同表述
  - 交叉引用追踪遗漏，前文提到的数据后文引用时不一致

### 机制 2: 四维审核
- **位置**: `core/audit/engine.py`
- **四个维度**:
  1. 结构合规（`structural.py`）— 文档结构是否匹配模板
  2. 引用有效性（`reference.py`）— 引用的标准/法规是否存在且未废止
  3. 语义合规（`semantic.py`）— 逐要点检查内容是否符合规则
  4. 内部一致性（`consistency.py`）— 文档内部是否自相矛盾
- **常见实现错误**:
  - 审核要点未关联到具体规则库条款，导致检查无据可依
  - 语义合规检查时 LLM prompt 未注入原文证据段落
  - 一致性检查只检查相邻章节，未做全文交叉检查

### 机制 3: 知识库检索（qmd）
- **位置**: `core/knowledge/store.py`
- **三个子库**: 案例库（按章节切片）、规则库（条款级粒度）、模板库
- **常见实现错误**:
  - 案例切片粒度不当（太粗 → 噪声多，太细 → 丢失上下文）
  - 规则库未处理版本/废止状态，召回已废止法规
  - namespace 隔离不严格，跨项目数据泄露

## 3. 配置与接入规范

### 项目配置文件
每个接入 Scrivai 的项目需要提供 `scrivai-project.yaml`，定义：
- 项目 namespace
- 知识库路径（案例/规则/模板）
- 章节定义（生成引擎用）
- 审核要点集（审核引擎用）
- 报告模板

### 接入测试检查清单
- [ ] 新项目仅通过配置文件接入，未修改 `core/` 下任何代码
- [ ] 知识库 namespace 与其他项目隔离
- [ ] 审核要点集覆盖该项目的所有规则要求
- [ ] 章节模板定义完整（每章有规格说明）
- [ ] E2E 测试通过（生成 + 审核 + 生成+自审 三条 pipeline）

## 4. 代码规范检查项
<!-- 通用检查清单，一般不改 -->

### 4.1 必须通过（阻塞合并）
- [ ] 函数签名有完整类型注解（`Union`, `Dict`, `Optional` 等）
- [ ] 模块、类、方法有中文 Docstring（功能、参数、返回值）
- [ ] 使用项目日志系统，无 `print()` 调试输出
- [ ] 命名规范：PascalCase 类名，描述性变量名，`_` 前缀私有变量
- [ ] 导入顺序：标准库 → 第三方 → 项目内部
- [ ] Commit message 符合 Conventional Commits，无 AI 标识

### 4.2 应该遵守（非阻塞但需标记）
- [ ] 使用阶段化注释组织复杂逻辑（`# Phase 1`, `# Phase 2`）
- [ ] 复杂接口返回值包含诊断信息
- [ ] 配置项在 config.yaml 中而非硬编码

## 5. 工程正确性检查（替代"科研正确性"）

### 5.1 生成质量检查（生成引擎相关代码必查）
- [ ] 上下文管理器正确维护术语表和交叉引用
- [ ] 前文摘要在每章生成后更新
- [ ] 章节依赖关系被正确处理（依赖章节先生成）
- [ ] LLM prompt 包含所有必要上下文（不遗漏、不超长）
- [ ] 生成后一致性校验确实执行

### 5.2 审核准确性检查（审核引擎相关代码必查）
- [ ] 审核要点与规则库条款正确关联
- [ ] 每个审核判定附带原文引用证据
- [ ] 引用有效性检查包含版本/废止状态判断
- [ ] 审核报告格式符合配置的模板

### 5.3 知识库正确性检查
- [ ] 入库时的切片逻辑正确（案例按章节、规则按条款）
- [ ] 检索结果的相关性合理（有集成测试验证）
- [ ] namespace 隔离严格，无跨项目泄露
- [ ] 元数据完整（来源文件、章节编号、版本信息等）

### 5.4 可配置性检查（新功能必查）
- [ ] 新功能通过配置驱动，不硬编码项目特定逻辑
- [ ] 配置 schema 有明确的类型定义和校验
- [ ] 默认配置合理，新项目接入不需要设置所有字段

---

## 6. 集成测试审查

- [ ] 生成引擎：至少一个完整的多章节生成测试，验证连贯性
- [ ] 审核引擎：至少覆盖四个维度各一个测试用例
- [ ] 编排层：三条 pipeline（只审/只生/生成+自审）各一个 E2E 测试
- [ ] 多项目：至少两个不同配置的项目跑通 E2E，验证通用性

```markdown

- 改动类型: [新功能/bug修复/重构/...]
- 关联模块: [知识库/生成引擎/审核引擎/编排层/SDK]

  - 为什么是问题: [影响]
  - 建议修复: [怎么改]
  - 相关规范: [§几]

## 📊 检查清单

[一句话：✅ 可合并 / ⚠️ 修复后可合并 / ❌ 需重做]
```
